## TaskBoard とは

```
Order から班ごとの Task を自動導出し、
現場の調整判断に必要な情報を整理して提示することを狙ったアプリです。
```

## 作ろうと思った理由

```
タイミーとして、
下処理、カット、検品、冷蔵庫への搬入・搬出、梱包、ライン作業など
様々な現場を経験させていただきました。

その中で、
Task の優先順位付けや人員配置の判断など、
現場の方の負担が大きそうに見える場面がありました。



もちろん、私が見られたのは御社全体の一部であり、
原因が社内システムにあると断定できる立場ではありません。

ただ、僭越ながら
「現場での判断を、社内システムで支えられる余地があるのでは」
と感じました。



そこで、
面談の機会をいただいたことをきっかけに、
現場で感じた課題を形にするつもりで TaskBoard を制作しました。
```

## 課題として意識した点

-   タイミーとして班を移動した際、
    指示どおり動いても行き先が変わることがあり、
    全体状況の共有や把握が難しい場面があるのではと感じた

-   出荷時間が迫る中で、
    優先順位の相談や調整が発生している場面を見聞きし、
    判断材料が早く揃うほど調整負担を下げられる可能性があると感じた

※私は現場の一部しか見られておらず、原因が社内システムにあると断定する意図はありません

## 苦労した点

一番苦労したのは、タイミーの立場では実際の社内システムや細かい運用が見えない中で、
記憶と想像を頼りに「現場の動きに合う設計」へ落とし込むことでした。

そのため、

-   数量をどの単位で管理するか（g / 袋 / ケース / 本）と
-   いつ・どこで・誰が・どう入力するか

の 2 点を何度も作り直しました。



迷走はありましたが、

-   現場の動きと合わない設計は採用しない
-   入力負担を増やさず、前より楽になったと感じられる形を優先する

という軸は崩さないようにしました。

## 主要な設計判断（結論）

-   数量の基本単位は「袋」を採用 (運用の軽さと必要十分な正確さを優先)

-   進捗記録のタイミングは「1 本」を採用 (入力頻度を抑えつつ把握可能)

-   入力は「スキャン中心 + ラベル不良時は手入力」のハイブリッド想定

-   端末配置は冷蔵庫の出入口付近を想定 (動線から外れにくいため)

-   端末台数は「初期は共用 1 台、将来的に班ごと」まで想定

## 今後実装したいこと

-   工場全体の Task の進捗やボトルネックを一覧できる画面
-   Task の優先順位ロジック
-   人員配置の判断材料を提示するロジック
-   社員・各班が Task の優先順位を画面上で調整できるようにする
-   各班が Task の進捗（出来高）を登録し、指示数に対してリアルタイムで残数を調整できるようにする
